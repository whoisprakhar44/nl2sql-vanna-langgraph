PART 1 — WHAT THIS SYSTEM IS (AND ISN'T)
================================================================================

This is NOT a Vanna application.
This is NOT a RAG (Retrieval-Augmented Generation) chatbot.
This is NOT a template-matching SQL generator.

This IS an agentic, reasoning-first NL-to-SQL system where:
  - The LLM (via LangGraph) does all the thinking
  - Vanna is just a search tool for finding relevant schema pieces
  - SQL is generated by reasoning, not by copying past examples

Think of it like a human analyst:
  - First they look up the relevant tables (Step 2 — retrieval)
  - Then they understand the schema (Step 3 — schema linking)
  - Then they write the SQL (Step 4 — generation + execution)
  - Then they interpret the results (Step 5 — analytics)

PART 2 — TECHNOLOGY STACK
================================================================================

VANNA 0.7.9
  - The LAST version using the legacy VannaBase API (before the 2.0 rewrite)
  - We use it ONLY to store DDL + documentation in ChromaDB and search them
  - Its SQL generation methods are explicitly BLOCKED in our code
  - If Vanna releases 0.8 or 1.0 one day, they don't exist — skip to 2.0
  - Key Vanna classes: ChromaDB_VectorStore, Ollama
  - Key Vanna methods: train(), get_related_ddl(), get_related_documentation(),
    get_similar_question_sql()

CHROMADB (>=0.4, <1.0)
  - A local vector database that stores text as mathematical vectors
  - When you "train" Vanna, it converts DDL/docs into vectors and stores them here
  - When you query, it finds the most similar vectors to your question
  - Data is persisted on disk at: data/chroma_store/
  - Has 3 collections: ddl_collection, documentation_collection, sql_collection

OLLAMA (local)
  - Runs LLMs locally on your Mac — no internet or API key needed
  - Used for: creating embeddings (converting text to vectors for ChromaDB)
  - Model on this machine: qwen3.5:2b (2.3B parameter model, fast)
  - URL: http://localhost:11434

PYDANTIC (>=2.0)
  - Python library for data validation with type annotations
  - All data models (SchemaContext, TableInfo, etc.) are Pydantic models
  - Gives us: type checking, JSON serialisation, validation, IDE autocomplete

PYDANTIC-SETTINGS
  - Extension of Pydantic for reading environment variables / .env files
  - The Settings class in src/config/settings.py uses this
  - Changes to .env are automatically reflected without code changes

PYYAML
  - Reads YAML files — used for the metadata files in data/metadata/
  - YAML is the format for describing each table's schema and business rules

LANGGRAPH (>=0.2)
  - Google's framework for building stateful, multi-step AI agent pipelines
  - Not yet used in the code — installed now for when Steps 3-5 are built
  - Works as a directed graph where each "node" is an agent or function
  - Think of it like a flowchart where each box can call an LLM

LANGCHAIN-OLLAMA
  - Connects LangChain/LangGraph to Ollama models
  - Will be used in Steps 3-5 for the reasoning agents

PART 4 — THE HYBRID RETRIEVAL STRATEGY EXPLAINED
================================================================================

When a question comes in, two retrieval systems run in parallel:

SEMANTIC RETRIEVAL (via Vanna + ChromaDB)
  - Converts the question to a vector using Ollama embeddings
  - Finds the top N vectors in ChromaDB that are most similar
  - Works even when the question uses different words than the schema
  - Example: "show sales data" → finds "CREATE TABLE orders" because
    the trained documentation says orders are purchase records
  - Weakness: may miss exact column names, can be noisy

KEYWORD RETRIEVAL (via MetadataStore + YAML)
  - Extracts meaningful words from the question (strips stop words)
  - Exact-match against table names, column names, descriptions
  - Works perfectly for technical terms mentioned exactly in the question
  - Example: "orders.status" → finds orders table and status column immediately
  - Strength: precise, fast, no embedding needed

THEN: FK RELATIONSHIP EXPANSION
  - After both systems find tables, we walk the FK graph
  - Any table connected to a found table via FK is automatically included
  - Example: found "orders" → automatically add "customers" (via customer_id FK)
  - This prevents the classic failure of missing join tables

THEN: MERGE AND DEDUPLICATE
  - Keyword results take priority (they have full YAML metadata)
  - Semantic results add any extra tables not caught by keyword match
  - Duplicates removed by table name

RESULT: SchemaContext with only the most relevant entities (3-5 tables max)

PART 5 — HOW STEPS 3-5 WILL CONNECT (FUTURE)
================================================================================

STEP 3 — Schema Linker Agent (LangGraph node)
  Input:  SchemaContext from Step 2
  Task:   Validate tables exist, resolve column ambiguity, confirm FK paths
  Output: Verified SchemaContext (possibly enriched with extra columns)
  Tool:   fetch_yaml (reads YAML metadata for any missing table)

STEP 4 — SQL Generation + Execution Agent (LangGraph node with tool loop)
  Input:  Verified SchemaContext from Step 3
  Task:   Generate SQL, execute it, auto-fix errors (max 3 iterations)
  Tools:  execute_sql, fetch_yaml
  Output: DataFrame result + the final SQL
  Loop:   If execution fails, send error back to LLM → regenerate → retry

STEP 5 — Analytics Agent (LangGraph node)
  Input:  DataFrame from Step 4
  Task:   Calculate stats, detect anomalies, compare to previous period,
          choose the right chart type
  Tools:  calculate_stats, detect_anomalies, compare_periods, pick_visualization
  Output: Formatted insight + chart specification
